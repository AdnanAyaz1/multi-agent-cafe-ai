# Coding Practices

The rules below are binding. Every PR / change follows them. New rules are
added here, not invented on the fly.

---

## 1. Component architecture (the core rule)

| Concern | Lives in |
| --- | --- |
| Data fetching, async work | **Page** (Server Component) |
| UI layout & composition | **Page / Component** |
| State + event handlers | **Client Component** |
| State *logic* (useState, useEffect, reducers) | **Reusable hook** (`hooks/`) |
| Pure business logic / calculations | **`utils/`** |
| Constants, enums, magic strings | **`constants/`** |
| Raw shadcn primitives | **`components/ui/`** (shadcn output) |
| Composed, reusable UI blocks | **`components/`** |

**Rule of thumb:** if a `.tsx` file contains `useState`, `useEffect`, or
`onClick`, it is a Client Component and starts with `'use client'`. The
*logic* of that state lives in a hook; the file is mostly JSX.

---

## 2. Pages are Server Components by default

```tsx
// app/page.tsx — server component
import { getTodaysRecommendation } from '@/lib/recommendation'

export default async function HomePage() {
  const data = await getTodaysRecommendation()
  return <Dashboard data={data} />
}
```

- No `'use client'` at the top of pages unless there's a hard reason.
- Data is fetched at the top of the page and passed down as props.
- Client islands (`'use client'`) are pushed as deep into the tree as
  possible — never the whole page.

---

## 3. Client components are thin

A Client Component is a **shell** that:
1. Calls a custom hook for state + side effects.
2. Renders JSX from primitives + composed components.

```tsx
'use client'
// components/chat-input.tsx
import { useChatInput } from '@/hooks/use-chat-input'
import { Button } from '@/components/ui/button'
import { Textarea } from '@/components/ui/textarea'

export function ChatInput() {
  const { value, setValue, submit, isSubmitting } = useChatInput()
  return (
    <form onSubmit={submit}>
      <Textarea value={value} onChange={(e) => setValue(e.target.value)} />
      <Button type="submit" disabled={isSubmitting}>Send</Button>
    </form>
  )
}
```

- No business logic in the JSX file.
- No `fetch`, no `JSON.parse`, no date math — those go in `utils/`.
- No hardcoded strings/arrays for filters, options, etc.

---

## 4. Hooks own the state logic

```ts
// hooks/use-chat-input.ts
'use client'
import { useState, useCallback } from 'react'
import { sendMessage } from '@/lib/actions'

export function useChatInput() {
  const [value, setValue] = useState('')
  const [isSubmitting, setIsSubmitting] = useState(false)

  const submit = useCallback(async () => { /* ... */ }, [])

  return { value, setValue, submit, isSubmitting }
}
```

- One hook per Client Component (or per logical concern).
- Hooks are the only place that calls Server Actions / API routes
  (the component just invokes the returned function).

---

## 5. Business logic in `utils/`

Pure functions only. No React, no DOM, no `fetch`.

```ts
// utils/recommendation.ts
export function categorizeItem(name: string): 'hot' | 'cold' | 'neutral' {
  /* ... */
}

export function applyDiscount(price: number, percent: number): number {
  return +(price * (1 - percent / 100)).toFixed(2)
}
```

- Pure = same input → same output, no side effects.
- Easy to unit-test.

---

## 6. Constants in `constants/`

```ts
// constants/filters.ts
export const CATEGORY_FILTERS = ['all', 'hot', 'cold', 'food', 'dessert'] as const
export type CategoryFilter = (typeof CATEGORY_FILTERS)[number]
```

```ts
// constants/agents.ts
export const AGENT_ROLES = ['planner', 'analyst', 'strategist', 'critic', 'synthesizer'] as const
```

- No magic strings or arrays in components.
- `as const` + derived type for enums.

---

## 7. UI: shadcn/ui + composed components

- shadcn primitives live in `components/ui/*` (generated by the CLI, do
  not edit unless necessary).
- Composed, project-specific UI lives in `components/*` and is built
  *only* from shadcn primitives + `utils/`.
- Styling: Tailwind utility classes only. No inline `style={}` unless
  dynamic and unavoidable.
- Accessibility comes free with shadcn — keep it (label every input, use
  `aria-*` where needed).

---

## 8. Next.js 16 specifics

- All request-time APIs are async: `await cookies()`, `await params`,
  `await searchParams`.
- Use the globally-typed `RouteContext<'/path/[id]'>` and `PageProps<...>`
  helpers (generated by `next dev` / `next typegen`).
- Server-only code is marked with `import 'server-only'`. Never import
  DB / Ollama / secrets code into a Client Component.
- Use the bundled docs at `node_modules/next/dist/docs/` before writing
  any Next.js code (per `AGENTS.md`).

---

## 9. TypeScript

- `strict: true` (already in `tsconfig.json`).
- No `any` in committed code. Use `unknown` + narrow.
- Prefer `type` for shapes, `interface` only when extension is needed.
- All exported functions have explicit return types.

---

## 10. Errors & logging

- Agent / network errors are typed (`OllamaError`, `WeatherError`).
- The UI shows a friendly error with a "what to try" hint.
- The server logs include: agent name, duration, status, error cause.
- No `console.log` left in production paths — use the logger from
  `utils/logger.ts`.

---

```
  api/

  agents.ts

  weather.ts
  menu-source.ts

```

---

## 12. Pull / change checklist (before any commit)

- [ ] Page is a Server Component (or justified Client Component).
- [ ] Client Components have `'use client'` and delegate logic to a hook.
- [ ] No business logic in `.tsx` files.
- [ ] No hardcoded arrays/strings — they live in `constants/`.
- [ ] No `any`, no `console.log` left behind.
- [ ] Server-only modules are guarded with `server-only`.
- [ ] `npm run lint` passes.
